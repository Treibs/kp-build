/kp-build · orchestration

How the skill works, step by step

Building one verified knowledge package is a pipeline across three kinds of worker: a human who scopes it, a fleet of AI agents that do the judgment (find, read, extract, refute, answer), and a deterministic engine that does the mechanical part (verify, assemble, score). Two gates make it trustworthy: a verification gate that drops invented citations, and an acceptance gate that proves it helps.

The three workers

Who does what — and why the seam matters

Judgment is delegated to agents; anything that must be exact and repeatable is delegated to the engine. The engine can't be talked into accepting a fake citation.

HumanYou

Scope the narrow research area at the start, and read the delivered package + report at the end. Two touch-points — everything between is autonomous.

AI agentsThe judgment

Survey the literature, read papers, extract claims + open problems, try to refute claims, synthesize, and write the two falsification answers. Dispatched in parallel, in isolation.

EngineThe mechanical core

kp_build — verifies every citation against arXiv/Crossref, assembles the package, lints it, and scores the falsification. Deterministic; the trust anchor.

The pipeline at a glance

Seven stages across three lanes

Stages 2–4 fan out into many parallel agents. Stages 5–6 are the two trust gates (green): the engine verifies citations before anything ships, and falsification proves the package beats an unaided agent before it's delivered.

Step by step

The orchestration in full

1

Scope Human · you

You and the orchestrator agree a narrow research area (one paper-sized topic, e.g. "speculative decoding for LLM inference"), the sub-questions a related-work section must answer, and what's in vs out of scope. This is the one upfront human step — it defines "done."

in: a research goalout: scope contract
2

Survey AI agents · fan-out

The orchestrator dispatches a survey agent per sub-question. Each does two passes: (a) keyword search (arXiv, Semantic Scholar, the venues) for the seed papers, then (b) citation-graph expansion — kp-build expand pulls the references and citers of the verified spine from Semantic Scholar, surfacing the seminal and newest papers a keyword search misses. Agents relevance-filter the candidates and keep only those with a real arXiv id / DOI.

in: scope contractout: real papers + handles
Discipline: a paper with no verifiable handle is dropped, never invented. Coverage is recorded so the gap is honest.
3

Extract · Ground · Refute AI agents · per paper, isolated

For each paper, an extraction agent pulls claims (each tied to a verbatim passage, scoped tightly to it), the open problems the paper flags as unsolved (the highest-value output), and any debate position. Then a separate refuter agent tries to break each high-stakes claim using the beat's other sources.

in: papersout: claims · open problems · positions · refuter verdicts
Isolation: the refuter sees only the claim and the other sources — never the drafter's reasoning. A claim it breaks is marked survived_refuter:false, and later shown capped + flagged, not silently dropped.
4

Synthesize AI agents

Cross-paper assembly: open problems that several papers flag are merged (corroboration that they're real), opposing positions are clustered into a debate map, and claims multiple independent papers support are marked corroborated_by. The output is one structured research.json.

in: per-paper extractionsout: research.json
5

Assemble Engine Verification gate

kp-build build takes the research JSON and verifies every citation live against arXiv and Crossref (throttled so large packages don't trip rate limits). A paper that doesn't resolve — or whose id points at a different paper — is dropped, and its claims with it. Only verified-anchored nodes ship. The engine then writes the package, the agent-loadable CONTEXT.md, the knowledge.json contract, and lints it.

in: research.jsonout: verified package + CONTEXT.md
This is the no-hallucination guarantee. It's mechanical and unbribable — an invented citation cannot pass, no matter how confident the agent that drafted it was.
6

Falsify Agents + Engine Acceptance gate

The acceptance test: two agents answer the same held-out task — one with no package (unaided recall), one loaded with CONTEXT.md. The engine (kp-build falsify) scores both on citation precision (do the cited ids resolve and match the paper named?) and recall (how much of the verified spine they used), and records an f1 verdict in the manifest.

in: package + a held-out taskout: HELPS / TIE / NO — a number
If the loaded agent doesn't beat the unaided one, the package isn't adding value — the skill says so plainly and either picks a more model-weak topic or deepens the survey. No hollow wins.
7

Deliver Engine → Human

Falsify is a prerequisite of the report — kp-build report refuses to render a package that hasn't been measured. You receive the package, the human-readable report (verdict, verified spine, open problems, debates), and CONTEXT.md for any agent to load. The same build also emits a knowledge.json, so the package is shareable through the open 0xLT/kpm manager — build once, anyone reuses it.

in: measured packageout: report + shareable package
